# Эксперимент: TLS, RFC 8879 (сжатие сертификатов), стенд VirtualBox — Ideco NGFW + ALT Workstation

Документ описывает **воспроизводимую** методику для курсовой: две уже развёрнутые ВМ (Ideco NGFW 21, Enterprise demo; ALT Linux Workstation), сеть, настройки на Ideco, пакеты и команды на ALT, **какие метрики снимать** и сколько повторов делать.

---

## 0. Важно понять до начала

**RFC 8879** действует только при **TLS 1.3**. Сжатие включается только если:

1. **Клиент** в `ClientHello` отправил расширение `compress_certificate` (тип **27**) со списком алгоритмов (zlib / brotli / zstd).  
2. **Сервер** поддерживает один из них и вместо сообщения `Certificate` (тип **11**) отправляет **`CompressedCertificate`** (тип **25**).

Сценарий **«без сжатия»** в эксперименте: клиент **не** отправляет расширение 27 **или** сервер не умеет сжимать — тогда на проводе остаётся обычное сообщение `Certificate`.

Сценарий **«с каждым алгоритмом»**: сервер и клиент оба должны уметь выбранный алгоритм. Проще всего получить контроль, подняв **тестовый TLS‑сервер на ALT** (`openssl s_server` с флагом сжатия сертификатов), а Ideco использовать как **маршрутизатор/МЭ/при необходимости reverse proxy** и точку **захвата `tcpdump`**.

*Публичные сайты в интернете часто уже сжимают сертификаты или наоборот — без вашего контроля нельзя гарантировать отдельный прогон «только zlib». Поэтому в методике ниже — **контролируемый сервер на ALT**.*

## 1. Топология VirtualBox (рекомендуемая)

Задайте так, чтобы трафик от ALT к тестовому порту на ALT **логически проходил через Ideco** (для курсовой это важно как «на базе Ideco NGFW»):

| ВМ | Роль | Пример интерфейсов |
|----|------|---------------------|
| Ideco NGFW | шлюз периметра, LAN, WAN (по вашей схеме) | LAN, например `192.168.1.1/24` |
| ALT Workstation | клиент + **тестовый TLS‑сервер** | IP в LAN, например `192.168.1.10`, **шлюз по умолчанию = LAN Ideco** |

**Тестовый сервер** слушает на ALT, например `0.0.0.0:8443` (HTTPS/TLS 1.3). Клиентские замеры идут с **этой же ВМ ALT** на `192.168.1.10:8443` (через стек и маршрут до «своего» адреса через шлюз — см. примечание ниже) **или** с отдельного пользовательского процесса; главное — в отчёте зафиксировать **точный маршрут** (прямой на localhost vs через шлюз).

**Примечание про «через Ideco»:** если хост обращается к **своему же** IP в той же подсети, часто трафик **не пойдёт** на шлюз (прямая доставка в ОС). Варианты для честного прохода через Ideco:

1. **Две сетевые карты ALT не обязательны:** поднять сервер на `192.168.1.10:8443`, а замеры делать с **другой маленькой ВМ** «клиент» в той же LAN (если появится третья ВМ).  
2. **Без третьей ВМ:** на Ideco настроить **DNAT / reverse proxy** с внешнего/LAN‑адреса Ideco на `192.168.1.10:8443`; тогда с ALT вызываете **`https://<IP_Ideco_LAN>/`** и пакеты идут Ideco → ALT (Ideco в пути).  
3. **Минимальный вариант для отчёта:** сервер и клиент на ALT, замеры по `127.0.0.1:8443` — Ideco вне пути; тогда в курсовой явно пишете: «контролируемая пара клиент–сервер на ALT; Ideco — подготовка политик и **дублирующий** захват при публикации через прокси». Выберите один вариант и **не смешивайте** в одной таблице разные маршруты.

Ниже в шагах предполагается **вариант 2** (прокси на Ideco на бэкенд ALT:8443) как наиболее соответствующий «на базе Ideco». Если прокси настроить нельзя — используйте вариант 1 или 3 и опишите в отчёте.

## 2. Пакеты на ALT Linux Workstation

Установите (имена пакетов типичны для ALT; при отличии найдите аналоги через `apt-cache search` / веб‑интерфейс репозитория):

```bash
su -
# или sudo — по принятой у вас схеме

apt-get update
apt-get install -y openssl curl tcpdump wireshark-cli ca-certificates
```

**Обязательно проверьте версию OpenSSL:**

```bash
openssl version -a
```

Для встроенного в `openssl` управления **сжатием сертификатов** в `s_client` / `s_server` нужна ветка **OpenSSL 3.2.0 и новее** (флаги вида `-cert_comp`). Если у вас **3.0.x / 1.1.1**:

- либо подключите репозиторий/пакет с OpenSSL ≥ 3.2, если есть для вашей версии ALT;  
- либо используйте **контейнер** (например, образ с Fedora 40+ и `openssl` 3.2+) только для команд `s_server`/`s_client` — это чуть больше работы на ALT, зато воспроизводимо.

Проверка наличия опции:

```bash
openssl s_client -help 2>&1 | grep -i cert_comp
openssl s_server -help 2>&1 | grep -i cert_comp
```

Если строк с `cert_comp` нет — обновляйте OpenSSL или см. раздел 8 (альтернатива GnuTLS).

## 3. Подготовка на ALT: сертификат и тестовый сервер

Рабочий каталог, например `~/tls-rfc8879-lab`.

### 3.1. Минимальная цепочка сертификатов (чтобы было что сжимать)

```bash
mkdir -p ~/tls-rfc8879-lab && cd ~/tls-rfc8879-lab

# корневой УЦ (демо)
openssl genrsa -out ca.key 4096
openssl req -x509 -new -nodes -key ca.key -sha256 -days 30 -subj "/CN=Demo RFC8879 CA" -out ca.crt

# серверный ключ и CSR
openssl genrsa -out server.key 2048
openssl req -new -key server.key -subj "/CN=test-rfc8879.local" -out server.csr

# серверный сертификат от своего CA
openssl x509 -req -in server.csr -CA ca.crt -CAkey ca.key -CAcreateserial -out server.crt -days 30 -sha256

# цепочка для сервера: leaf + intermediate (здесь только leaf + CA как «цепочка» для демо)
cat server.crt ca.crt > chain.pem
```

Для **более тяжёлой** цепочки (сильнее заметно сжатие) добавьте второй промежуточный сертификат тем же способом и включите в `chain.pem` — по желанию для курсовой.

### 3.2. Запуск `openssl s_server` (пример порта 8443)

Сервер **без** сжатия сертификатов (базовый режим):

```bash
cd ~/tls-rfc8879-lab
openssl s_server -accept 8443 -www -tls1_3 -cert server.crt -key server.key -CAfile ca.crt
```

Сервер **с** сжатием, например **zstd** (повторите отдельными сессиями для **zlib**, **brotli**):

```bash
# zstd
openssl s_server -accept 8443 -www -tls1_3 -cert server.crt -key server.key -CAfile ca.crt -cert_comp zstd

# zlib
openssl s_server -accept 8443 -www -tls1_3 -cert server.crt -key server.key -CAfile ca.crt -cert_comp zlib

# brotli
openssl s_server -accept 8443 -www -tls1_3 -cert server.crt -key server.key -CAfile ca.crt -cert_comp brotli
```

Оставьте сервер в отдельном терминале. Для каждой серии замеров перезапускайте сервер с нужным `-cert_comp` (или без него).

## 4. Настройка Ideco NGFW (что сделать на шлюзе)

Лицензия Enterprise demo уже есть — ниже только сетевая логика и сервисы. Точные пункты меню сверяйте с [документацией v21](https://docs.ideco.ru/v21.pdf).

### 4.1. Сеть и маршрутизация

- Убедитесь, что **ALT в LAN** получает адрес (DHCP или статика), **шлюз по умолчанию = LAN‑адрес Ideco**.  
- DNS на ALT: либо Ideco как DNS, либо публичный — зафиксируйте в отчёте (влияет на `time_namelookup`).

### 4.2. Межсетевой экран

- Разрешите **из LAN на ALT:8443** TCP (если клиент ходит на ALT напрямую).  
- Если используете **reverse proxy** на Ideco (вариант из раздела 1):  
  - разрешите **из LAN на IP Ideco (порт публикации, например 443)**;  
  - на Ideco в правилах **прокси / публикации** укажите бэкенд `192.168.1.10:8443` (подставьте ваши IP/порт).

Цель: не блокировать тестовый TLS.

### 4.3. Reverse proxy (рекомендуется для «трафик через Ideco»)

В веб‑интерфейсе Ideco (раздел публикации / reverse proxy — имя по v21.pdf):

- Внешний (или LAN) **виртуальный хост** HTTPS, например `https://192.168.1.1/` или отдельное имя, если настроите DNS.  
- Бэкенд: `https://192.168.1.10:8443` с проверкой сертификата бэкенда **или** `http` если переведёте сервер на plain (для курсовой лучше end‑to‑end TLS до ALT).

**Внимание:** некоторые прокси **терминируют** TLS на Ideco и устанавливают **новое** TLS к бэкенду. Тогда сжатие сертификата между **клиентом и Ideco** — это свойство TLS‑стека Ideco, а не вашего `s_server`. Для строгого контроля RFC 8879 между клиентом и **вашим** `openssl s_server` либо:

- публикуйте TCP **прозрачно** (TCP‑прокси без TLS на Ideco), если такая опция есть,  
- либо в отчёте разделите сценарии: «A: клиент ↔ Ideco», «B: клиент ↔ ALT напрямую».

Зафиксируйте в отчёте, какой вариант выбрали.

### 4.4. Захват трафика на Ideco (для Wireshark)

Если в консоли Ideco / SSH доступен `tcpdump` (как в учебных материалах по продукту):

```bash
# пример: интерфейс LAN — подставьте имя из ip link / документации
tcpdump -i LAN_INTERFACE -w /tmp/tls-rfc8879.pcap 'host 192.168.1.10 and tcp port 8443'
```

Или фильтр по порту публикации на Ideco. Файл `.pcap` скопировать на ALT/хост и открыть в Wireshark.

## 5. Клиент на ALT: сценарии «без сжатия» и «с каждым алгоритмом»

Все замеры выполняйте при **уже запущенном** соответствующем `s_server`.

### 5.1. Базовая проверка рукопожатия (openssl)

**Без предложения сжатия со стороны клиента:**

```bash
openssl s_client -connect 192.168.1.10:8443 -tls1_3 -servername test-rfc8879.local -CAfile ~/tls-rfc8879-lab/ca.crt </dev/null
```

**С предложением сжатия (пример zstd)** — синтаксис уточните по `openssl s_client -help`:

```bash
openssl s_client -connect 192.168.1.10:8443 -tls1_3 -servername test-rfc8879.local -CAfile ~/tls-rfc8879-lab/ca.crt -cert_comp zstd </dev/null
```

Аналогично замените `zstd` на `zlib` и `brotli` для трёх алгоритмов RFC 8879.

В выводе ищите признаки успешного TLS 1.3; при включённой трассировке (если доступно в вашей сборке):

```bash
openssl s_client -connect ... -tls1_3 ... -trace -msg 2>&1 | tee handshake.txt
```

(Опции `-trace`/`-msg` зависят от сборки.)

### 5.2. Соответствие сервера режиму клиента

| Эксперимент | Запуск `s_server` | Клиент `s_client` |
|-------------|-------------------|-------------------|
| **1. Без сжатия (контроль)** | **без** `-cert_comp` | **без** `-cert_comp` → ожидайте обычный `Certificate` |
| **2. Zstd** | `-cert_comp zstd` | `-cert_comp zstd` (и при необходимости перечислите несколько, если man допускает) |
| **3. Zlib** | `-cert_comp zlib` | `-cert_comp zlib` |
| **4. Brotli** | `-cert_comp brotli` | `-cert_comp brotli` |

Если клиент предлагает сжатие, а сервер запущен без `-cert_comp`, согласования сжатия **не будет** — это нормальный контрольный случай.

## 6. Метрики: что снимать и как фиксировать

Для каждой комбинации (строка таблицы ниже) делайте **3–5 повторов** (в задании — 3–5; для курсовой можно **5**).

### 6.1. Метрики уровня приложения (curl)

`curl` ходит по HTTPS к `s_server` с режимом `-www` (отдаёт простейший ответ). Подставьте URL вида `https://IP:8443/` (или URL через Ideco).

Файл `~/tls-rfc8879-lab/curl-format.txt`:

```text
time_namelookup:  %{time_namelookup}\n
time_connect:     %{time_connect}\n
time_appconnect:  %{time_appconnect}\n
time_pretransfer: %{time_pretransfer}\n
time_starttransfer: %{time_starttransfer}\n
time_total:       %{time_total}\n
```

**Основная метрика для курсовой (время TLS после TCP):**

\[
T_{\mathrm{TLS}} = \texttt{time\_appconnect} - \texttt{time\_connect}
\]

Дополнительно (для анализа):

- `time_namelookup` — влияние DNS (при IP почти ноль).  
- `time_starttransfer - time_appconnect` — до первого байта ответа после TLS.

**Серия из 5 прогонов:**

```bash
URL="https://192.168.1.10:8443/"   # подставьте свой URL (через Ideco — другой)
for i in 1 2 3 4 5; do
  echo "=== run $i ==="
  curl -k -sS -o /dev/null -w "@$HOME/tls-rfc8879-lab/curl-format.txt" "$URL"
done | tee -a curl-runs-control.txt
```

Повторите для каждого сценария (без сжатия; zlib; brotli; zstd), **не меняя** между сериями нагрузку на ВМ (закройте лишние приложения).

**Записать в таблицу:**

| Сценарий | Алгоритм RFC 8879 | Повтор | time_connect | time_appconnect | T_TLS | time_total |
|----------|-------------------|--------|--------------|-----------------|-------|------------|
| Контроль | нет | 1…5 | … | … | … | … |
| Сжатие | zlib | 1…5 | … | … | … | … |
| Сжатие | brotli | 1…5 | … | … | … | … |
| Сжатие | zstd | 1…5 | … | … | … | … |

После серий — **среднее**, **медиана**, при возможности **p95** по пяти значениям \(T_{\mathrm{TLS}}\).

### 6.2. Метрики уровня протокола (Wireshark по pcap)

На одном прогоне на сценарий сохраните pcap (с ALT или с Ideco):

- Есть ли в `ClientHello` расширение **compress_certificate (27)** и какие коды алгоритмов (1 / 2 / 3).  
- В ответе сервера: тип сообщения **11** (`Certificate`) или **25** (`CompressedCertificate`).  
- Размер **суммарных** TCP‑полезных нагрузок до `Finished` (грубо: фильтр по потоку, оценка по Expert или экспорт I/O graph — по возможности).

Занесите в отчёт **скриншот** или таблицу «сценарий → тип сообщения → да/нет расширение 27».

### 6.3. Метрики CPU (опционально)

На ALT во время 5 прогонов:

```bash
top -b -n 1 | head
```

Или `pidstat` если установлен. Цель — показать, что при brotli/zlib CPU может отличаться от zstd (качественно).

## 7. Порядок работ в день эксперимента (чеклист)

1. Запустить Ideco и ALT, проверить связность `ping` с ALT на шлюз и на IP сервера.  
2. На ALT: установить пакеты, проверить `openssl version` и `-cert_comp`.  
3. Сгенерировать/проверить сертификаты в `~/tls-rfc8879-lab`.  
4. **Серия A (контроль):** `s_server` **без** `-cert_comp` → 5× `curl` → записать метрики → 1× pcap + разбор Wireshark.  
5. **Серия B:** `s_server -cert_comp zlib` + клиент с `zlib` → 5× `curl` → pcap.  
6. **Серия C:** `brotli` — то же.  
7. **Серия D:** `zstd` — то же.  
8. (Если настроен прокси на Ideco) повторить минимум серии A и D **через URL Ideco**, отдельная таблица.  
9. Заполнить сводную таблицу средних/медиан и вывод: где виден `CompressedCertificate`, насколько отличается \(T_{\mathrm{TLS}}\) при вашей сети (RTT мал — разница может быть в пределах погрешности).

## 8. Альтернатива, если на ALT нет OpenSSL 3.2+

Установите `gnutls-cli` и при наличии опции сжатия сертификатов (в GnuTLS поддержка RFC 8879 появлялась в развитых версиях; проверьте `gnutls-cli --help`):

```bash
apt-get install -y gnutls-bin
gnutls-cli --help | grep -i compress
```

Если опции есть — методика аналогична: сервер с поддержкой + клиент с `--compress-cert=...`, повторы 3–5, те же метрики через `curl` к совместимому серверу.
